# سند نیازمندی و درخواست پیشنهاد اجرای پروژه (RFB)
## WorkPilot – سیستم مدیریت پروژه، وظایف و گزارش‌دهی سازمانی

**نسخه:** 1.0

---

## 1. معرفی پروژه

**نام محصول:** WorkPilot

**نوع محصول:** سامانهٔ تحت وب مدیریت پروژه، وظایف، گزارش‌دهی و تحلیل عملکرد تیم‌ها

**هدف پروژه:**
طراحی و توسعهٔ یک پلتفرم یکپارچه برای سازمان‌ها و تیم‌ها به‌منظور مدیریت پروژه‌ها، کنترل وظایف، ثبت فعالیت‌های کاری، بررسی عملکرد و تولید گزارش‌های مدیریتی.

این سیستم با هدف کاهش وابستگی به فایل‌های پراکنده، گزارش‌های دستی و ابزارهای غیرمتمرکز طراحی می‌شود.

---

## 2. اهداف اصلی سیستم

سیستم باید امکان موارد زیر را فراهم کند:

- مدیریت پروژه‌های سازمان
- تعریف و پیگیری وظایف
- تخصیص کار به اعضای تیم
- ثبت فعالیت‌های روزانهٔ کارکنان
- بررسی و تأیید گزارش‌ها
- تحلیل عملکرد تیم
- تهیهٔ گزارش‌های مدیریتی
- خروجی گرفتن از اطلاعات سیستم
- طراحی واکنش‌گرا (Responsive) که تجربهٔ کاربری کامل را در گوشی موبایل نیز فراهم کند

---

## 3. محدودهٔ پروژه (Scope)

### در محدودهٔ فاز اول:
- پنل وب مدیر سازمان، مدیر پروژه و کارمند
- مدیریت پروژه، وظیفه و زیروظیفه
- ثبت گزارش کاری و فرآیند تأیید
- داشبورد مدیریتی و گزارش‌گیری
- خروجی فایل (Excel، PDF، CSV)
- اعلان‌های داخل‌سیستمی و ایمیلی

### خارج از محدودهٔ فاز اول (موکول به فازهای بعدی):
- اپلیکیشن موبایل
- دستیار هوش مصنوعی و تحلیل پیش‌بین
- ادغام با ابزارهای شخص‌ثالث (Slack، Jira، تقویم گوگل و…)
- مدل صورت‌حساب/پرداخت برای فروش SaaS چندمشتریه (در صورت نیاز، جداگانه مشخص شود)

## 4. کاربران و نقش‌های سیستم

> تصمیم معماری: سیستم از همان ابتدا **چندسازمانی (Multi-tenant SaaS)** طراحی می‌شود؛ چند سازمان مستقل روی یک زیرساخت مشترک با مدل Shared-Schema و ستون `organization_id` برای ایزوله‌سازی داده هر سازمان.

### مدیر پلتفرم (Platform Admin)
امکانات:
- مدیریت سازمان‌های ثبت‌شده در سیستم
- نظارت کلی بر وضعیت سرویس
- مدیریت تنظیمات سطح پلتفرم

### مدیر سازمان
امکانات:
- مدیریت کاربران سازمان
- ایجاد و مدیریت پروژه‌ها
- مشاهدهٔ وضعیت پروژه‌ها
- بررسی عملکرد کارکنان
- تأیید یا رد گزارش‌ها
- دریافت گزارش‌های مدیریتی

### مدیر پروژه
امکانات:
- مدیریت پروژه‌های مربوطه
- تعریف وظایف
- تخصیص وظایف به اعضا
- بررسی میزان پیشرفت
- بررسی گزارش‌های کاری

### کارمند
امکانات:
- مشاهدهٔ وظایف اختصاص‌داده‌شده
- ثبت فعالیت‌های انجام‌شده
- ثبت زمان صرف‌شده برای فعالیت‌ها
- اعلام میزان پیشرفت کار
- ارسال گزارش برای بررسی

---

## 5. امکانات اصلی سیستم

### 5.1 مدیریت سازمان و کاربران
- ایجاد فضای کاری برای هر سازمان
- مدیریت اعضای سازمان (دعوت، غیرفعال‌سازی، حذف)
- تعریف نقش‌های کاربری
- کنترل سطح دسترسی کاربران

### 5.2 مدیریت پروژه‌ها
- ایجاد پروژهٔ جدید
- تعیین تاریخ شروع و پایان
- تعیین اعضای پروژه
- مشاهدهٔ وضعیت پروژه
- مشاهدهٔ میزان پیشرفت پروژه

### 5.3 مدیریت وظایف (Task Management)
- ایجاد وظیفه
- تخصیص وظیفه به افراد
- تعیین اولویت
- تعیین Deadline
- تعیین وضعیت انجام کار
- **وابستگی بین وظایف** (وظیفهٔ B پس از تکمیل وظیفهٔ A)
- **پیوست فایل** به وظیفه
- **کامنت/گفتگو** روی هر وظیفه برای هماهنگی تیم
- نمای فهرستی، **نمای کانبان (Kanban)** و **نمودار گانت** برای پیگیری بصری

ساختار وظایف به‌صورت سلسله‌مراتبی:
```
پروژه
 └─ وظیفهٔ اصلی
     └─ زیروظیفه
```

### 5.4 سیستم ثبت گزارش کاری
کاربران می‌توانند:
- فعالیت انجام‌شده را ثبت کنند
- زمان صرف‌شده را وارد کنند
- میزان پیشرفت را مشخص کنند
- توضیحات کاری اضافه کنند

### 5.5 سیستم تأیید گزارش‌ها
فرآیند بررسی گزارش:

```
کارمند → ارسال گزارش → بررسی مدیر → تأیید یا درخواست اصلاح
```

### 5.6 داشبورد مدیریتی
نمایش اطلاعات:
- تعداد پروژه‌ها
- وضعیت پروژه‌ها
- میزان پیشرفت
- ساعات کاری ثبت‌شده
- عملکرد اعضای تیم
- فعالیت‌های اخیر

### 5.7 گزارش‌گیری پیشرفته
سیستم امکان گزارش‌گیری بر اساس موارد زیر را دارد:
- کاربر
- پروژه
- وضعیت
- بازهٔ زمانی

امکان انتخاب بازه:
- روزانه
- هفتگی
- ماهانه
- بازهٔ دلخواه

### 5.8 خروجی فایل
گزارش‌ها قابلیت خروجی گرفتن دارند:
- **Excel** — برای تحلیل و مدیریت اطلاعات
- **PDF** — برای گزارش‌های رسمی
- **CSV** — برای انتقال داده به سیستم‌های دیگر

> نکتهٔ فنی: برای گزارش‌های حجیم، تولید فایل باید به‌صورت غیرهمزمان (Async / Background Job) انجام و لینک دانلود یا اعلان آماده‌شدن فایل به کاربر ارسال شود تا رابط کاربری قفل نشود.

### 5.9 اعلان‌ها
سیستم قابلیت اطلاع‌رسانی برای موارد زیر را خواهد داشت:
- ایجاد وظیفهٔ جدید
- نزدیک شدن زمان تحویل
- ارسال گزارش
- تأیید یا رد گزارش

**کانال‌های اعلان:** درون‌برنامه‌ای (In-app) به‌صورت پیش‌فرض؛ ایمیل به‌صورت اختیاری/قابل‌تنظیم توسط کاربر.

## 7. امنیت سیستم

سیستم شامل موارد زیر خواهد بود:

- ورود امن کاربران (رمزنگاری رمز عبور با الگوریتم استاندارد، امکان فعال‌سازی احراز هویت دومرحله‌ای (2FA))
- سیاست رمز عبور (حداقل طول، پیچیدگی، انقضا/عدم انقضا)
- مدیریت نشست (Session) با انقضای خودکار پس از عدم فعالیت
- مدیریت سطح دسترسی مبتنی بر نقش (RBAC)
- محافظت از اطلاعات سازمان‌ها و **ایزوله‌سازی کامل داده بین سازمان‌های مختلف** در مدل چندسازمانی
- رمزنگاری ارتباط (HTTPS/TLS) و رمزنگاری داده‌های حساس در حالت ذخیره‌سازی (At Rest)
- ثبت فعالیت‌های کاربران (Audit Log) به‌صورت غیرقابل تغییر
- کنترل دسترسی به اطلاعات بر اساس نقش و مالکیت داده

## 8. مدل دادهٔ خلاصه (Data Model Overview)

موجودیت‌های اصلی:
- **Organization** (سازمان)
- **User** (کاربر) — با نقش و متعلق به یک سازمان مشخص (مدل چندسازمانی از ابتدا؛ به‌جز نقش مدیر پلتفرم که سراسری است)
- **Project** (پروژه)
- **Task / SubTask** (وظیفه/زیروظیفه)
- **WorkLog / Report** (گزارش کاری/فعالیت)
- **Attachment** (پیوست فایل)
- **Comment** (کامنت روی وظیفه)
- **Notification** (اعلان)

## 9. معماری کلی سیستم

سیستم به‌صورت یک نرم‌افزار تحت وب توسعه داده می‌شود.

**اجزا و استک فنی انتخاب‌شده:**

### بخش کاربری (Frontend)
React + TypeScript + Vite + Tailwind CSS، با کامپوننت‌های shadcn/ui؛ رابط کاربری وب واکنش‌گرا (Mobile-First) برای کاربران و مدیران.

### بخش پردازشی (Backend)
FastAPI (Python) با SQLAlchemy ORM؛ مدیریت:
- منطق سیستم
- کاربران
- پروژه‌ها
- وظایف
- گزارش‌ها
- امنیت
- Job های ناهمزمان (Celery + Redis) برای تولید خروجی فایل حجیم

### پایگاه داده
PostgreSQL برای ذخیرهٔ امن اطلاعات (کاربران، پروژه‌ها، وظایف، گزارش‌ها) و Redis برای کش/صف کار.

### استقرار
Docker / docker-compose برای بسته‌بندی و اجرای یکپارچهٔ سرویس‌ها (backend، frontend، پایگاه داده، Redis).

## 12. معیارهای پذیرش پروژه (Acceptance Criteria)

پروژه زمانی تکمیل‌شده تلقی می‌شود که:
- تمامی امکانات مندرج در بخش ۵ پیاده‌سازی و در محیط آزمایشی قابل تست باشند
- نیازمندی‌های غیرعملکردی بخش ۶ محقق شده باشند
- سیستم تست امنیتی (حداقل بررسی OWASP Top 10) را با موفقیت پشت سر گذاشته باشد
- مستندات فنی (نصب/استقرار) و راهنمای کاربری تحویل داده شده باشد
- کارفرما آموزش استفاده از سیستم را دریافت کرده باشد
- رابط کاربری در حداقل یک مرورگر موبایل (iOS Safari / Android Chrome) و در سایزهای صفحهٔ کوچک تست و تأیید شده باشد

## 13. ریسک‌ها و مفروضات

**مفروضات:**
- زیرساخت میزبانی (ابری/داخلی) توسط کارفرما یا مجری بر اساس توافق جداگانه تأمین می‌شود.

**ریسک‌های شناسایی‌شده:**
| ریسک | اثر | راهکار کاهش |
|---|---|---|
| تغییر دامنهٔ کار (Scope Creep) پس از شروع توسعه | تأخیر و افزایش هزینه | مدیریت تغییرات رسمی (Change Request) |
| حجم بالای داده در گزارش‌گیری | کندی سیستم | Async Job برای خروجی فایل‌های حجیم |
