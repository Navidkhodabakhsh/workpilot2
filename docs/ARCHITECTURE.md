## چندمستأجری (Multi-tenant)

- مدل: **Shared-Schema** — یک دیتابیس مشترک، هر جدول تننت‌محور دارای ستون `organization_id`.
- ایزوله‌سازی در لایهٔ سرویس اعمال می‌شود: یک dependency مشترک (`get_current_org_scope`) که `organization_id` کاربر جاری را استخراج و در تمام کوئری‌های سرویس اعمال می‌کند.
- نقش `platform_admin` از این محدودیت مستثناست (دسترسی سراسری برای مدیریت پلتفرم).
- در آینده در صورت نیاز به ایزوله‌سازی سخت‌تر، می‌توان به Row-Level Security پستگرس یا schema-per-tenant مهاجرت کرد؛ این تصمیم فعلاً باز گذاشته نشده و به عنوان مسیر ارتقا مستند است.

## Backend (FastAPI)

```
backend/app/
  core/        → config (pydantic-settings), security (JWT, hashing), dependencies (current_user, RBAC, org scope)
  db/          → SQLAlchemy engine/session, Alembic migrations
  models/      → Organization, User, Project, Task, TaskDependency, WorkLog, Comment, Attachment, Notification, AuditLog
  schemas/     → Pydantic request/response
  api/routers/ → auth, organizations, users, projects, tasks, worklogs, reports, dashboard, notifications, exports
  services/    → business logic + RBAC rules per-endpoint
  workers/     → Celery tasks (Redis broker) برای تولید ناهمزمان خروجی فایل
backend/tests/ → pytest
```

نقش‌ها (enum روی `User.role`): `platform_admin`, `org_admin`, `project_manager`, `employee`.

### پیاده‌سازی سوییت خودکار (فاز I)

- **بک‌اند:** `backend/tests/conftest.py` یک دیتابیس Postgres جدا (`workpilot_test`) می‌سازد و به هر تست یک `Session` واقعی می‌دهد که سرویس‌ها روی آن `commit()` صدا می‌زنند (نه تراکنش rollback-based) — این عمداً انتخاب شد چون رفتار commit خودِ سرویس‌ها بخشی از چیزی است که باید تست شود؛ ایزولاسیون بین تست‌ها با `TRUNCATE ... CASCADE` روی همهٔ جدول‌ها بعد از هر تست تأمین می‌شود.
- **نکتهٔ حیاتی برای هر تغییر آینده در `app/db/session.py`:** `engine` در سطح ماژول و در زمان import ساخته می‌شود، از `settings.database_url`. بنابراین `conftest.py` باید `os.environ["DATABASE_URL"]` را **قبل از اولین import** (مستقیم یا غیرمستقیم) از `app.db.session` تنظیم کند؛ در غیر این صورت Celery worker (که session خودش را با `SessionLocal()` جدا می‌سازد، نه session تست را به اشتراک می‌گذارد) به دیتابیس dev وصل می‌شود و بی‌صدا داده‌ای پیدا نمی‌کند — دقیقاً همین باگ یک‌بار در توسعهٔ فاز I رخ داد (به Handoff فاز I در `PROJECT_STATE.md` مراجعه کنید).
- Celery با `task_always_eager=True` + `task_eager_propagates=True` در تست‌ها اجرا می‌شود تا Job های async (خروجی فایل) بدون یک worker process جدا، هم‌زمان با درخواست HTTP اجرا شوند.
- **فرانت:** vitest (نه Jest) چون از همان پیکربندی Vite/esbuild پروژه استفاده می‌کند و نیازی به تنظیمات جدا ندارد؛ تست‌ها به‌جای mock کردن رفتار واقعی، روی توابع خالص (`cn`, `notificationMessage`)، state واقعی (`useAuthStore`) و رندر واقعی کامپوننت (`Button` با `@testing-library/user-event`) نوشته شدند.
- **CI:** `.github/workflows/ci.yml` دو Job مستقل دارد — `backend` (با سرویس‌های Postgres و Redis روی همان runner) و `frontend` (typecheck + build + vitest) — روی هر push و pull request. Playwright هنوز در CI اجرا نمی‌شود (فقط دستی در طول توسعه استفاده شد) چون نیاز به هم‌زمان بالا آوردن بک‌اند+فرانت در runner دارد؛ افزودنش به‌عنوان کار آیندهٔ اختیاری باقی می‌ماند.

## استقرار

`docker-compose.yml` با سرویس‌های: `postgres`, `redis`, `backend` (uvicorn), `worker` (celery), `beat` (celery beat), `frontend` (vite dev / build production).

### مهاجرت دیتابیس در Docker (فاز J)

`backend/docker-entrypoint.sh` قبل از اجرای فرمان اصلی کانتینر (`uvicorn`/`celery`)، در صورتی که متغیر محیطی `RUN_MIGRATIONS=true` تنظیم شده باشد، `alembic upgrade head` را اجرا می‌کند. این متغیر فقط برای سرویس `backend` در `docker-compose.yml` تنظیم شده — نه `worker`/`beat` — تا سه پردازه هم‌زمان روی یک دیتابیس تازه (روی اولین `docker compose up`) با هم race نکنند. `worker` و `beat` با `depends_on: backend: condition: service_started` منتظر شروع کانتینر backend می‌مانند (نه تضمین قطعی پایان migration، ولی در عمل کافی است چون migration اولیه چند ثانیه طول می‌کشد).

## تصمیم‌های ریز اما پرتبعات (طراحی دیتابیس، امنیت، API)

> این تصمیم‌ها با سطح بالای استدلال گرفته شدن چون تغییرشان بعداً پرهزینه است.

### مدل داده و چندمستأجری
- **یکتایی ایمیل کاربر: سراسری (Global)، نه در حد سازمان.** ساده‌تر برای فرآیند ورود (نیازی به انتخاب/تشخیص سازمان در لاگین نیست). محدودیت شناخته‌شده: یک ایمیل نمی‌تواند در دو سازمان مختلف حساب داشته باشد؛ در صورت نیاز آینده به ورود یک نفر به چند سازمان، باید جدول Membership جداگانه اضافه شود (مسیر ارتقا، نه نیاز فعلی).
- **`organization_id` روی جداول فرزند (Task، WorkLog، Comment، Attachment، Notification، AuditLog) دنرمالایز می‌شود**، حتی وقتی از طریق `project_id` قابل استخراج است. دلیل: دفاع در عمق در برابر نشت داده بین سازمان‌ها — فیلتر مستقیم بدون JOIN، و جلوگیری از باگ «فراموش‌کردن اسکوپ» در سرویس‌ها.
- **ایزوله‌سازی چندمستأجری در سطح کد، نه دیتابیس:** یک کلاس پایه `TenantScopedRepository` تمام کوئری‌ها را با `organization_id` فیلتر می‌کند؛ هیچ سرویسی مستقیم به session خام دسترسی نمی‌گیرد. مسیر ارتقای آینده در صورت نیاز به سخت‌گیری بیشتر: Row-Level Security پستگرس.
- **وابستگی بین وظایف (TaskDependency):** تشخیص چرخه (cycle) در سطح سرویس انجام می‌شود، نه constraint دیتابیس (چون DB این محدودیت را به‌سادگی پشتیبانی نمی‌کند).

### احراز هویت و امنیت
- **JWT دوتکه:** access token کوتاه‌عمر (۳۰ دقیقه) حاوی `user_id`، `organization_id`، `role`؛ refresh token بلندمدت (۷ روز) در کوکی httpOnly+Secure (نه localStorage) تا ریسک سرقت توکن با XSS کم شود.
- **هش رمز عبور:** bcrypt از طریق passlib.
- **RBAC:** بررسی نقش به‌صورت dependency factory (`require_role(*roles)`) در سطح روتر، نه شرط پراکنده داخل توابع.
- **Rate limiting روی ورود** برای مقابله با brute-force در فاز H اضافه می‌شود؛ از همین فاز A ساختار dependency injection طوری طراحی می‌شود که افزودنش بعداً نیازی به بازنویسی نداشته باشد.

### ساختار API
- REST با پیشوند نسخه: `/api/v1/...`.
- بدون envelope اضافه دور پاسخ‌ها — مستقیماً مدل‌های Pydantic (سادگی به‌جای انتزاع زودهنگام).
- Pagination: offset/limit ساده (نه cursor) — مقیاس فعلی این پروژه نیازی به cursor pagination ندارد.
- خطاها: `HTTPException` استاندارد FastAPI با فیلد `detail`.

### الگوی نرم‌افزاری
لایه‌بندی: **Router (HTTP) → Service (منطق کسب‌وکار + مجوز) → Repository/Model (دسترسی داده)**. این لایه‌بندی برای پیچیدگی RBAC چندمستأجری این پروژه توجیه دارد و انتزاع زائد نیست.

| عملیات | org_admin | project_manager | employee |

| ساخت پروژه | همه‌جا | بله (خودش عضو می‌شود) | ✗ |
| دیدن/ویرایش پروژه | همهٔ پروژه‌های سازمان | فقط پروژه‌هایی که عضوشان است | فقط پروژه‌هایی که عضوشان است (فقط دیدن) |

| ساخت/ویرایش/حذف وظیفه | همه‌جا | فقط پروژه‌های خودش | ✗ |
| تغییر وضعیت وظیفهٔ خودش | — | — | بله (فقط فیلد status) |

نکتهٔ پیاده‌سازی: سازندهٔ یک پروژه به‌طور خودکار عضو آن می‌شود تا `project_manager` بلافاصله بتواند آن را مدیریت کند. تشخیص چرخه در وابستگی وظایف با DFS روی گراف `depends_on` در لایهٔ سرویس انجام می‌شود (نه constraint دیتابیس).

`POST/GET /api/v1/users` (مدیریت کاربران سازمان توسط org_admin) در فاز C اضافه شد؛ در RFB بخش ۴.۱ پیش‌بینی شده بود ولی در فاز B از قلم افتاده بود.

## خروجی فایل ناهمزمان (فاز F)

- `ExportJob` وضعیت job را نگه می‌دارد (`pending`→`processing`→`done`/`failed`)؛ فایل تولیدشده روی دیسک مشترک backend/worker ذخیره می‌شود (`backend/exports/`، در `.gitignore`).
- Celery task کاربر درخواست‌دهنده را از دیتابیس بارگذاری می‌کند و همان تابع `services.reports.query_worklog_report` را با آن کاربر صدا می‌زند — یعنی RBAC چندمستأجری دقیقاً همان چیزی‌ست که در API اعمال می‌شود، نه یک مسیر جدا.
- **دسترسی به Job:** فقط سازندهٔ همان Job یا `org_admin` — نه هر عضو سازمان — چون یک گزارش می‌تواند دادهٔ چند پروژه/کاربر را همزمان شامل شود که ممکن است از دید یک `project_manager`/`employee` دیگر پنهان باشد.

## سخت‌سازی امنیتی (فاز H)

- **Rate limiting ورود:** کلید محدودیت بر اساس ایمیل است، نه IP — چون در استقرارهای واقعی پشت پراکسی/Load Balancer معمولاً همهٔ کاربران از دید سرور یک IP مشترک دارند؛ محدودیت به‌ازای ایمیل مستقیماً حساب هدف حمله را محافظت می‌کند. ۵ تلاش در ۵ دقیقه، با Redis (`INCR`+`EXPIRE`).
- **Refresh Token:** کوکی httpOnly+SameSite=Lax با `path` محدود به `/api/v1/auth` (نه کل دامنه). در هر استفاده rotate می‌شود (توکن قدیمی عملاً بی‌اثر می‌شود چون یک توکن جدید جایگزینش می‌کند). `cookie_secure` در تنظیمات پیش‌فرض `False` است (برای dev روی http)؛ در production باید `True` شود وگرنه مرورگر کوکی را روی HTTPS نادیده می‌گیرد در حالت `Secure`.
- **Audit Log:** فقط رویدادهای امنیتی/حساسِ منتخب ثبت می‌شوند (signup، login، تأیید/رد گزارش کار، حذف وظیفه) — نه هر mutation. هیچ مسیر update/delete برای این جدول تعریف نشده؛ خودِ همین نبود مسیر، تضمین‌کنندهٔ «غیرقابل تغییر بودن» است.
- **سیاست رمز عبور:** حداقل ۸ کاراکتر + حداقل یک حرف + حداقل یک رقم (`schemas/validators.py`، مشترک بین همهٔ مسیرهای ساخت کاربر).

## بازطراحی بصری و تکمیل منو (فازهای K تا R)

پس از اولین اجرای واقعی پروژه توسط کاربر (خارج از این محیط)، بازخورد نشان داد ۸ از ۱۱ آیتم منو صرفاً stub بودند و طراحی بصری (فونت/رنگ/سایدبار) هرگز واقعاً از skill نصب‌شدهٔ `ui-ux-pro-max` استفاده نکرده بود. جزئیات کامل در Handoff فاز K تا R در `docs/PROJECT_STATE.md`؛ تصمیم‌های معماری‌ای که برای آیندهٔ پروژه اهمیت دارند:

- **پالت رنگی از skill، نه دلبخواه:** با اجرای واقعی `scripts/search.py` روی دیتابیس CSV skill، ردیف "Productivity Tool" (teal/orange) انتخاب و در همان معماری سه‌لایهٔ توکن موجود (`index.css`) جایگزین شد — فقط مقدار primitive ها عوض شد، ساختار لایه‌ها دست‌نخورده ماند. Accent (نارنجی) عمداً در هیچ کامپوننت پیش‌فرضی wire نشده، فقط `variant="accent"` روی دکمه برای یک CTA اصلی در هر صفحه.
- **نکتهٔ حیاتی RTL برای هر توسعهٔ آینده روی چیدمان:** خواص منطقی CSS (`start-*`/`end-*` در Tailwind) نسبت به `direction` خودِ المان resolve می‌شوند، نه نسبت به فرزند اول یک flex container. سایدبار دسکتاپ چون فقط با ترتیب DOM (نه position) روی راست قرار می‌گیرد، درست بود؛ ولی drawer موبایل که با `end-0` (absolute) پوزیشن‌دهی شده بود، تحت این RTL به `left:0` resolve می‌شد — یعنی از سمت مخالف سایدبار دسکتاپ باز می‌شد. همیشه با اندازه‌گیری واقعی DOM (نه فرض) این نوع باگ را تأیید کنید.
- **سه ماژول بدون تعریف در RFB (گردش کار/پیام‌ها/تقویم) با تفسیر آگاهانه و مستند ساخته شدند**، نه با حدس بی‌مستند: گردش کار = Kanban سراسری (نه پروژه‌محور)، پیام‌ها = نظرات سطح وظیفه با مدل `Comment` موجود (نه چت)، تقویم = گرید مهلت وظایف با محاسبهٔ میلادی + برچسب جلالی از `Intl.DateTimeFormat(..., {calendar:"persian"})` (بدون کتابخانهٔ تاریخ جدید).
- **Postgres enum نیاز به migration دارد، نه فقط تغییر کد پایتون:** افزودن `comment_added` به `NotificationType` نیازمند یک migration جدید با `ALTER TYPE ... ADD VALUE` بود؛ صرفاً اضافه‌کردن مقدار به enum پایتون کافی نیست چون SQLAlchemy این enum را به‌صورت native Postgres type می‌سازد.
- **الگوی ذخیرهٔ فایل کاربر (Attachment) از الگوی `exports_dir` کپی شد ولی یک لایهٔ ایزولاسیون بیشتر دارد:** فایل‌های خروجی در یک پوشهٔ مسطح ذخیره می‌شوند، ولی پیوست‌های کاربر (چون محتوای آپلودشده توسط کاربر نهایی‌اند، نه تولیدشدهٔ سیستم) در زیرپوشهٔ مجزای هر سازمان (`attachments/<org_id>/`) ذخیره می‌شوند — یک لایهٔ دفاعی اضافه، هرچند ایزولاسیون اصلی همچنان از طریق `organization_id` در دیتابیس تضمین می‌شود.

## ریبرندینگ Tadvin Hesab و بازسازی تسک‌ها/تقویم (فازهای S تا Z)

پس از تکمیل فاز K تا R، کاربر یک دور بازخورد بسیار مفصل داد که شامل ریبرندینگ کامل (نام و پالت رنگی)، اصلاحات UI متعدد، و بازسازی کامل دو ماژول «تسک‌ها» و «تقویم» بود که قبلاً با تفسیر ساده در فاز Q ساخته شده بودند. جزئیات کامل در Handoff فاز S تا Z در `docs/PROJECT_STATE.md`؛ تصمیم‌های معماری‌ای که برای آیندهٔ پروژه اهمیت دارند:

- **Task Status و Approval Status دو فیلد کاملاً مستقل‌اند، نه یک state machine واحد:** `TaskStatus` (`todo/in_progress/completed/archived`) نشان می‌دهد کار در چه مرحله‌ای است؛ `ApprovalStatus` (`pending/approved/rejected`, nullable) کاملاً جداگانه نشان می‌دهد آیا مدیر پروژه آن را تأیید کرده یا نه. یک وظیفه می‌تواند هم‌زمان Completed و Pending باشد. رسیدن به `completed` به‌صورت ضمنی `approval_status` را `pending` می‌کند؛ رد کردن (`reject`) وضعیت را به `in_progress` برمی‌گرداند و تأیید/رد فقط وقتی مجاز است که `approval_status` واقعاً `pending` باشد (جلوگیری از تأیید دوباره).
- **تغییر مقدار enum پستگرس (نه فقط افزودن) نیاز به الگوی متفاوتی از migration افزایشی `ADD VALUE` دارد:** کاهش `TaskStatus` از ۵ به ۴ مقدار (با تغییر نام `done`→`completed` و ادغام `in_review`→`in_progress`, `blocked`→`todo`) نیازمند ساخت یک enum type جدید، backfill ستون از طریق یک ستون موقت، و جایگزینی کامل type قدیم بود؛ برخلاف افزودن ساده یک مقدار به enum موجود (مثل `comment_added` در فاز K-R) که فقط یک `ALTER TYPE ... ADD VALUE` می‌خواهد.
- **تسک شخصی (بدون پروژه) به‌جای ساخت یک پروژهٔ مخفی برای هر کاربر:** `Task.project_id` به `nullable` تغییر کرد؛ وقتی خالی باشد یعنی تسک شخصی است و `assignee_id` همیشه باید برابر سازندهٔ آن باشد (اجباری در سرویس‌لایه، نه فقط قرارداد). تسک‌های شخصی نه گردش‌کار تأیید دارند، نه در Kanban سراسری («گردش کار») نمایش داده می‌شوند — فقط در تب «تسک‌های شخصی من» دیده می‌شوند.
- **`actual_hours` هرگز ذخیره نمی‌شود، همیشه محاسبه‌شده است:** از جمع `WorkLog.time_spent_minutes` با وضعیت `approved` برای همان تسک به‌دست می‌آید (همان الگوی تجمیعی که در `dashboard.py`/`reports.py` از قبل بود)، سپس به‌صورت یک attribute موقت (نه ستون mapped) روی شیء ORM قبل از serialize شدن با Pydantic نشانده می‌شود — چون Pydantic v2 با `from_attributes=True` مقدار پیش‌فرض فیلد را وقتی attribute واقعی روی شیء وجود نداشته باشد جایگزین می‌کند، این الگو بدون نیاز به ساخت دستی dict کار می‌کند.
- **`TaskActivityLog` یک AuditLog جداگانهٔ per-task است، نه استفادهٔ مجدد از AuditLog عمومی:** همان الگوی غیرقابل‌تغییر بودن (بدون تابع update/delete) ولی مخصوص یک تسک، چون باید در UI (دیالوگ جزئیات تسک) قابل نمایش جدا از audit log سراسری org_admin باشد. کامنت و آپلود فایل روی یک تسک هم یک ردیف فعالیت در همین جدول می‌سازند (نه فقط تغییرات status/assignee/priority).
- **ورود با ایمیل یا موبایل از طریق یک فیلد واحد (`identifier`)، نه دو فیلد جدا:** `LoginRequest.identifier` بر اساس وجود `@` مشخص می‌کند کدام ستون (`email` یا `phone_number`) باید جست‌وجو شود؛ کلید rate-limiting هم از `email` به `identifier` تغییر کرد. `phone_number` روی `User` یک ستون nullable و unique جداست، نه بخشی از `email`.
- **CalendarEvent قوانین دیدپذیری متفاوتی نسبت به الگوی استاندارد `get_visible_project_ids` دارد:** رویدادهای بدون پروژه (تعطیلی/جلسهٔ سراسری) برای همه قابل‌دیدن‌اند؛ مرخصی/یادآوری فقط برای سازنده، کاربر مرتبط، و project_manager (برای دید کلی روی در دسترس‌بودن تیم) قابل‌دیدن‌اند؛ org_admin همیشه همه‌چیز را می‌بیند. این قانون به‌طور آگاهانه از الگوی «فقط پروژه‌های عضو» که در تسک/پروژه استفاده می‌شود متفاوت است، چون رویدادهای تقویم لزوماً به یک پروژه وابسته نیستند.
- **FullCalendar (`@fullcalendar/react` v6) به‌جای گرید دستی، ولی همچنان با محاسبهٔ میلادی + برچسب جلالی:** انتخاب شد چون تعامل‌های درخواستی (Week/Day/Agenda، Drag & Drop، Resize) عملاً بازسازی یک تقویم کامل بود. Toolbar پیش‌فرض FullCalendar غیرفعال شد (`headerToolbar={false}`) و با دکمه‌های سفارشی این اپ جایگزین شد تا عنوان بازهٔ نمایش‌داده‌شده با `Intl.DateTimeFormat(..., {calendar:"persian"})` سازگار با بقیهٔ اپ بماند؛ فقط `locale="fa"` از خودِ FullCalendar برای متن‌های داخلی‌اش (مثل پیام «رویدادی نیست» در نمای برنامه) استفاده شد. مهلت وظایف به‌عنوان یک event source جداگانهٔ فقط‌خواندنی (`editable:false`) در کنار `CalendarEvent`های واقعی نمایش داده می‌شوند؛ Drag/Resize فقط برای `CalendarEvent` فعال است.
- **`TaskDetailDialog` به یک حالت controlled اختیاری مجهز شد بدون شکستن استفادهٔ قبلی:** پراپ‌های `open`/`onOpenChange` اختیاری اضافه شدند (وقتی داده نشوند، رفتار قبلی—state داخلی + `trigger`—دست‌نخورده می‌ماند)؛ این تغییر لازم بود چون تقویم باید بتواند این دیالوگ را از کلیک روی یک پین تسک به‌طور برنامه‌ای باز کند، نه از کلیک مستقیم روی یک trigger که در DOM تقویم وجود ندارد.

## احراز هویت موبایل‌محور با OTP (دور سوم بازخورد، پس از فاز Z)

پس از تست واقعی فازهای S تا Z، کاربر خواستار حذف کامل ایمیل از صفحهٔ ورود، ورود دومنظوره (رمز عبور یا کد یکبار مصرف پیامکی)، قابلیت فراموشی رمز، و رفع یک باگ P0 در پیام خطای ورود شد. جزئیات کامل در Handoff مربوطه در `docs/PROJECT_STATE.md`؛ تصمیم‌های معماری مهم:

- **`hashed_password` روی `User` عمداً nullable شد، نه یک ستون جدا مثل `has_completed_setup`:** `None` به‌تنهایی معنای «این کاربر با موبایل دعوت شده و هنوز رمز تعیین نکرده» را حمل می‌کند؛ هر تابعی که با رمز عبور مقایسه می‌کند (`authenticate`, `change_password`) باید صریحاً این حالت را قبل از `verify_password` رد کند تا کرش نکند (بررسی `hashed_password is None` قبل از `verify_password(x, None)`).
- **`OtpCode` بر اساس `phone_number` خام کلید می‌خورد، نه FK به `User`:** یک کد باید بتواند برای شماره‌ای درخواست شود که هنوز مشخص نیست دقیقاً کدام رکورد کاربر را resolve می‌کند (یا اصلاً هست یا نه) — گرچه در عمل این پروژه فقط اجازهٔ درخواست کد برای شماره‌های ثبت‌شده را می‌دهد (404 برای شمارهٔ ناشناس)، مدل داده عمداً کوپل به شناسهٔ کاربر نشد تا تغییر بعدی (مثلاً اجازهٔ signup مستقیم با موبایل) نیاز به migration نداشته باشد.
- **کد OTP دقیقاً مثل رمز عبور با bcrypt هش می‌شود، نه plaintext در دیتابیس** — حتی برای یک کد ۶رقمی کوتاه‌عمر، چون نشت دیتابیس نباید کد فعال کاربر را افشا کند؛ همان توابع `hash_password`/`verify_password` بازاستفاده شدند تا منطق هش تکرار نشود.
- **دو Purpose جدا (`login` در برابر `password_reset`) به‌جای یک نوع کد مشترک:** یک کد OTP که برای بازیابی رمز درخواست شده هرگز نباید بتواند مستقیم برای ورود هم استفاده شود (چون این دو مسیر پیامدهای امنیتی متفاوتی دارند — بازیابی رمز، رمز را عوض می‌کند). `verify_otp` این را با فیلتر صریح `purpose` روی کوئری تضمین می‌کند.
- **پرچم صریح `settings.sms_provider_configured` برای نبود SMS gateway واقعی در این محیط:** به‌جای mock کردن یک ارائه‌دهنده یا توقف کار برای پرسیدن، این پرچم (پیش‌فرض `False`) باعث می‌شود `request_otp` کد را مستقیم در پاسخ API برگرداند (`debug_code`)؛ این مسیر باید قبل از استقرار واقعی حذف/غیرفعال شود (کامنت صریح در کد + `docs/PROJECT_STATE.md` این را مستند می‌کند تا فراموش نشود).
- **Rate limiting دوگانه و مستقل:** درخواست کد (`otp_request:{purpose}:{phone}`، ۳ بار در ۱۰ دقیقه) از تلاش ورود (`login_attempts:{identifier}`، ۵ بار در ۵ دقیقه) کاملاً جدا نگه داشته شده، چون این دو محور حمله متفاوتی را محدود می‌کنند (spam پیامک در برابر حدس‌زدن کد/رمز). یک اثر جانبی این جداسازی: چون هر دو آستانه به‌طور مستقل روی این پروژه ۵ تنظیم شده‌اند، تستی که می‌خواهد صرفاً «قفل‌شدن کد بعد از حداکثر تلاش» را بسنجد باید مستقیماً سرویس (`services.otp.verify_otp`) را صدا بزند، نه از مسیر HTTP روتر که rate limiter سطح بالاتر زودتر 429 برمی‌گرداند.
- **صفحهٔ ورود دیگر هیچ فیلد ایمیل ندارد، ولی endpoint بک‌اند `LoginRequest.identifier` هنوز ایمیل را هم می‌پذیرد** (میراث فاز W) — این تصمیم آگاهانه است: ابزارهای `platform_admin` و اسکریپت‌های seed همچنان با ایمیل کار می‌کنند؛ فقط سطح UI کاربر نهایی محدود به موبایل شد، نه کل سطح API.

### باگ P0: پیام خطای گمراه‌کننده روی rate limit

صفحهٔ ورود برای هر شکست (۴۰۱ رمز اشتباه *و* ۴۲۹ محدودیت نرخ) دقیقاً یک پیام ثابت نشان می‌داد. کاربری که به سقف تلاش می‌رسید هیچ نشانه‌ای از قفل موقت نمی‌دید و مدام دوباره تلاش می‌کرد. رفع شد با شاخه‌بندی صریح `err.response.status === 429` در catch بلاک فرانت. **درس کلی برای هر صفحهٔ ورود/فرم حساس آینده:** هر کد وضعیت متفاوت از سرور باید پیام کاربرپسند مجزای خودش را داشته باشد؛ یکی‌کردن پیام خطا برای «سادگی» یا «امنیت» (جلوگیری از افشای اطلاعات) فقط باید عمداً و برای کدهای هم‌ارز امنیتی (مثل ۴۰۱ کاربر نامعتبر در برابر ۴۰۱ رمز اشتباه) انجام شود، نه برای هر کد وضعیتی که اتفاقی مشابه به‌نظر می‌رسد.
